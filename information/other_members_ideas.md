/content/（実行時のカレントディレクトリ）にpublic_150.jsonとinference.json（標準コード2を実行すると出てくる提出用のファイル）をアップロードしてから実行してください。

```ipynb

import json
import pandas as pd
from collections import defaultdict
import yaml
import toml
import xml.etree.ElementTree as ET
import csv
import io

# --- 1. バリデータ関数群 ---
def validate_format(text, fmt):
    try:
        if fmt == 'JSON':
            json.loads(text)
        elif fmt == 'YAML':
            yaml.safe_load(text)
        elif fmt == 'TOML':
            toml.loads(text)
        elif fmt == 'XML':
            ET.fromstring(text)
        elif fmt == 'CSV':
            # CSVは最低1行以上あるかチェック
            if not text.strip(): raise ValueError("Empty CSV")
            pd.read_csv(io.StringIO(text))
        else:
            return False, f"Unknown format: {fmt}"
        return True, None
    except Exception as e:
        return False, str(e)

# --- 2. メイン評価ロジック ---
def evaluate_with_task_id(public_file, inference_file):
    print(f"Loading ground truth: {public_file}")
    with open(public_file, 'r', encoding='utf-8') as f:
        ground_truth = json.load(f)

    # task_id -> output_type の辞書を作成
    id_to_type = {}
    for entry in ground_truth:
        tid = str(entry.get('task_id'))
        otype = entry.get('output_type')
        if tid and otype:
            id_to_type[tid] = otype.upper() # 大文字統一

    print(f"Loaded {len(id_to_type)} task definitions.")

    print(f"Loading inference: {inference_file}")
    with open(inference_file, 'r', encoding='utf-8') as f:
        inferences = json.load(f)

    # 集計用
    stats = defaultdict(lambda: {"total": 0, "valid": 0})
    errors = []

    for item in inferences:
        tid = str(item.get('task_id'))
        generated_text = item.get('generation', '') # 推論結果

        # task_idに対応する正解タイプを取得
        target_fmt = id_to_type.get(tid)

        if not target_fmt:
            print(f"⚠️ Warning: Task ID {tid} not found in ground truth.")
            continue

        # 評価実行
        is_valid, error_msg = validate_format(generated_text, target_fmt)

        stats[target_fmt]["total"] += 1
        if is_valid:
            stats[target_fmt]["valid"] += 1
        else:
            errors.append({
                "task_id": tid,
                "format": target_fmt,
                "error": error_msg,
                "generated_snippet": generated_text[:100] # 先頭100文字だけ記録
            })

    # --- 3. レポート出力 ---
    print("\n" + "="*50)
    print("🏆 STRICT EVALUATION REPORT (Based on task_id)")
    print("="*50)

    total_valid = 0
    total_count = 0

    for fmt, data in stats.items():
        rate = (data["valid"] / data["total"]) * 100 if data["total"] > 0 else 0
        print(f"[{fmt}]")
        print(f"  Success Rate: {rate:.1f}%  ({data['valid']} / {data['total']})")
        total_valid += data["valid"]
        total_count += data["total"]

    print("-" * 50)
    overall_rate = (total_valid / total_count) * 100 if total_count > 0 else 0
    print(f"OVERALL SCORE: {overall_rate:.2f}%")
    print("="*50)

    # エラー詳細の保存（デバッグ用）
    if errors:
        print(f"\nFound {len(errors)} errors. Saving to 'validation_errors.json'...")
        with open('validation_errors.json', 'w', encoding='utf-8') as f:
            json.dump(errors, f, indent=2, ensure_ascii=False)

# --- 実行 ---
evaluate_with_task_id('public_150.json', 'inference.json')
```

標準コードにはアップサンプリングの仕組みが組み込まれていますが、対象がdaichira系のためu-10beiには適用できません。structured_data_with_cot_dataset_512_v2などのu-10bei系にも対応できるように考えてみました。
アップサンプリングは、def apply_upsampling関数で行われています。以下は、daichiraを想定されていますので、u-10bei系に合わせます。u-10bei系では、"metadata"の"format"に"csv"や"toml"などの形式が格納されています。def apply_upsampling関数の以下の行を変更します。
```py
packs = train_ds["subcategory"] if "subcategory" in train_ds.column_names else [None]*len(train_ds) → packs = [m["format"] for m in train_ds["metadata"]]
```
※実装が間違えや効率的な方法があればご指摘ください。
os.environのところの設定を変更します。u-10beiでは、変換先の形式のみ指定します。{"toml": 2.0}では、TOML形式のデータを２倍に水増しします。formatは小文字です。json形式なので複数記載することができます。

```py
os.environ["SFT_USE_UPSAMPLING"] = "1" # "1" で有効, "0" で無効
os.environ["SFT_UPSAMPLE_RULES"] = '{"toml": 2.0}'
```
データについては、データを水増ししたうえで、np.random.choiceを用いて重複を許し元のデータ件数分抽出するようにコーディングされています。よって、学習用のデータ件数は増えません。
私はこの設定でたまたまかもしれませんが、提供いただいたツールで"TOML"の精度が、56%→70%に向上しました。
また、u-10bei系では、"metadata"の"type"により"generation"による生成と、"conversion"による変換があります。今回の評価は変換なので、データを"type"が"conversion"に絞る方法もあるかと思います。

# はじめに

HuggingFaceのtrlは、SFTやDPO以外にもさまざまな手法をサポートしています。また、さまざまなハイパーパラメータを設定することができます。

https://huggingface.co/docs/trl/index

DPOの標準コードを変更すれば簡単に試すことができます。何点か試してみました。

# DPOのloss_typeの変更

DPOのLossは、以下のようにさまざまな損失関数をサポートしています。複数の関数を組み合わせ設定することもできます。

https://huggingface.co/docs/trl/dpo_trainer#loss-functions

DPOConfigのloss_typeに"ipo"を設定してみました。DPOの損失関数はsigmoidのためchosenとrejectedの差が大きくなると勾配が0に近くなり学習が進まなくなります。

勾配消失に対応する損失関数がIPOです。対応方法です。DPOConfigに以下を追加

`loss_type="ipo",`

結果として精度が少し良くなりました。正確に対応しようとすると学習率などのハイパーパラメータの調整も必要になると思います。

DPOのドキュメントには、loss_typeの組み合わせ方法も掲載されているので試してみる価値はあると思います。

# CPO(Contrastive Preference Optimization)

DPO用に提供されたデータで学習可能な手法として、CPO(Contrastive Preference Optimization)とORPO(Odds Ratio Preference Optimization)があります。

CPOを試してみました。

https://huggingface.co/docs/trl/cpo_trainer

CPOは、翻訳時に適切であるだけでなく、完璧な翻訳ができるように学習するようです。今回のデータでいくと、文法的に正しいだけでなく中身についても完全になることを期待して試してみました。

DPOからの変更は難しくありません。TrainerとConfigをDPOからCPOに変更するのみです。

```
#from trl import DPOTrainer, DPOConfig
from trl import CPOTrainer, CPOConfig
# DPOConfig→CPOConfig,DPOTrainer→CPOTrainer
dpo_config = CPOConfig(
・・・
trainer = CPOTrainer(
・・・
```

基本、これだけ変更すれば動作します。

他にも損失関数として、SimPOを利用しました。長い文章が不利になるため長さで正規化し不利にならないようにします。

`loss_type="simpo",`

あと、DPOの学習率のままだと学習が進まなかったため調整を行いました。

結果として精度が改善しました。今回は試していませんが、ORPOもCPO同様に変更すれば試すことができます。trlでサポートしている手法であれば簡単に試せるので、ぜひ手法やハイパーパラメータを変更し試してみてください。

# Person Q

LLMコンペで気づいた「×判定」と「スコア」の関係
　ison提出前のチェッカーがいくつか紹介されていますね。（感謝）
私も、チェッカーを使ってみての学び（私見なので、参考程度にしてね）
があったので、共有します。

「×（不正解）が少ないのに、スコアが思ったほど伸びない」
 逆に、 「×が少し増えただけで、スコアが大きく落ちることもある」
これ、普通の“正解率”の感覚だと理解しにくいので、以下のように
整理します。

ポイント①：「×」は“内容が違う”ではなく“形式が壊れた”が多い
×判定の多くは、内容以前に

TOML / YAML / XML / JSON / CSV の形式（構文）が正しくない
1文字のミスでも機械的にアウトになり得る
というタイプです。
例：
YAMLのインデントが1段ずれる
TOMLで = や , の位置が不正

XMLで & をそのまま書いてしまう（&amp; が必要）
人間の目だと「ほぼ合ってる」に見えても、パーサは厳密なので×になります。

ポイント②：採点スコアは「正解率（〇の数÷150）」ではない
このコンペのスコアは、単純に〇が何問 / 150問では決まりません。
多くの場合、評価は

形式が正しいか（パース可能か）
型が合っているか（数値/文字/配列など）
階層が合っているか（ネスト構造）
指定されたキー・配列の位置が守られているか
のような「構造の一致度」をまとめて数値化しています。
そのため：

〇でも満点とは限らない
×でもケースによっては“部分点”が入ることがある（または逆に大きく減点される）
ということが起こるようです。

ポイント③：スコアに効くのは“内容”より“形式の安定”
おそらく、内容を頑張るより、まず形式ミスを潰す方がスコアに効く。
特にTOMLやYAMLは、ちょっとした書き方の違いで壊れやすい。

モデル（Public）
https://huggingface.co/beachcities/qwen3-4b-sft-v5g-hybrid-merged
モデルカードに技術的な詳細を記載しています。
最大のポイント: Empty Think Injection
Qwen3はデフォルトで <think>...</think> の思考出力を生成します。構造化データ出力タスクでは、このthinking内でコードフェンス（```json）や前置き文（"Here is the output:"）が混入し、パースエラーの原因になります。

解決策: SFTデータの全assistant出力を以下の形式に統一しました。

<think>
</think>

{生の構造化データ}やっていることは単純です：
元データのCoT（Approach: ... Output: ...）を正規表現で物理削除

空の <think> ブロックを先頭に付与
コードフェンス・前置き文・後書きも正規表現で除去
これにより、モデルはthinkingフェーズを即座に終了し、クリーンな構造化データだけを出力するようになります。

データセット構成（12,686件）
運営提供の9つのデータセットからルールベースの前処理のみで構築しています。

u-10bei/structured_data_with_cot_dataset_512_v4（4,608件）: 基礎データ
daichira/structured-hard-sft-4k（4,000件）: 高難度データ
Deep Structure Upsampling（3,578件）: daichira hard-4kから深い構造（XML depth≥4, YAML depth≥5）を3倍増幅
CSV Dot-Notation Expansion（500件）: daichira hard-4kの深いYAMLをPythonで平坦化し、CSVドット記法→深いJSON/YAMLの変換ペアを合成

スコア改善で学んだこと
PSR（パース成功率）の改善だけでなく、パース成功しているタスクの出力がどれだけ深い構造を正しく再現できているかも重要だと感じました。Deep Structure UpsamplingとCSV Dot-Notation Expansionは、深い階層の出力品質を高めるためのアプローチです。

学習設定

Unsloth + LoRA (r=64, alpha=64)
lr=5e-5, 2 epochs, batch=16
load_in_4bit=True
標準コードからの変更はSFT前処理部分のみ

    ## はじめに

    松尾研コンペの学習データ前処理コード（`data_cleaning.ipynb`）を共有します。

        ▼

        │  YAML: YAML 1.1 (Y1〜Y5)

        │

        │

    ④ 余計な文章付加チェック（説明文・コードフェンス境界）

    ## 各ステップの詳細

    ### ① 収集・統一フォーマット化

    運営提供の9データセットを統一形式に正規化して結合します。

    - **u-10bei系（6種）**: `metadata["format"]` から出力フォーマットを取得
    - **daichira系（3種）**: `category` 列（`C_TOML`, `C_XML` 等）からフォーマットを抽出
    - 出力フォーマットが特定できないレコード（`source_format == unknown`）はこの段階で除外

    ### ② フォーマット別構文検証

    各フォーマットの公式仕様に基づき、assistant出力の構文妥当性を検証します。単に「パースできるか」だけでなく、仕様レベルで定義された制約を個別にチェックしています。

    #### CSV (RFC 4180)

    | チェック | 内容 |
    |---------|------|
    | C1 | ヘッダー + 1行以上のデータ行が存在すること |
    | C2 | 全行のフィールド数がヘッダーと一致すること |
    | C3 | ダブルクォートの開閉が対応していること |
    | C4 | フィールド途中の裸のダブルクォートがないこと（`""` でエスケープ） |
    | C5 | カンマ・改行・クォートを含むフィールドが適切にクォートで囲まれていること |

    #### JSON (RFC 8259)

    | チェック | 内容 |
    |---------|------|
    | J1 | 空出力ではないこと |
    | J2 | `json.loads` でパース可能であること |
    | J3 | `NaN` / `Infinity` / `-Infinity` を含まないこと |
    | J4 | オブジェクトキーの重複がないこと |
    | J5 | トップレベルが object または array であること |

    #### YAML (YAML 1.1 / PyYAML SafeLoader)

    | チェック | 内容 |
    |---------|------|
    | Y1 | 空出力ではないこと |
    | Y2 | YAML構文としてパース可能であること |
    | Y3 | 空ドキュメント（`None`）でないこと |
    | Y4 | マッピングキーの重複がないこと |
    | Y5 | トップレベルが mapping または sequence であること |

    > PyYAMLの`SafeLoader`はYAML 1.1仕様に基づいています（`yes`/`no` 等がbooleanとして解釈される）。YAML 1.2厳密準拠が必要な場合は `ruamel.yaml` 等の利用を検討してください。

    #### XML (XML 1.0)

    | チェック | 内容 |
    |---------|------|
    | X1 | 空出力ではないこと |
    | X2 | XMLパーサでパース可能であること |
    | X3 | `<!DOCTYPE>` / `<!ENTITY>` を含まないこと（安全性のため） |
    | X4 | ルート要素のタグ名が有効であること |
    | X5 | トップレベルがXML要素として取得できること |

    #### TOML (TOML 1.0)

    | チェック | 内容 |
    |---------|------|
    | T1 | 空出力ではないこと |
    | T2 | `tomllib.loads` でパース可能であること |
    | T3 | 重複キー等の仕様違反がないこと |
    | T4 | 空白のみのドキュメントでないこと |
    | T5 | トップレベルが table（dict）であること |

    ### ③ 重複除去

    3段階のアプローチで重複を検出・除去します。

    **第1段階: 完全一致ハッシュ（フルキー）**
    - `フォーマット + 入力元フォーマット + ユーザー入力 + 正規化済み出力` のSHA-256ハッシュ
    - 完全に同一のサンプルを検出

    **第2段階: 完全一致ハッシュ（出力のみ）**
    - `フォーマット + 正規化済み出力` のSHA-256ハッシュ
    - 異なるプロンプトでも同じ出力を持つサンプルを検出

    **第3段階: SimHash近似重複検出**
    - 64bit SimHash + LSHバンディング（4バンド×16ビット）
    - ハミング距離3以内のペアを近似重複として除去

    重要な工夫として、ハッシュ計算前に**フォーマット別の正規化（canonicalization）**を行っています。

    - JSON: `json.dumps(sort_keys=True)` で意味的に同一なら同一ハッシュに
    - XML: `ET.tostring()` でDOMレベルで正規化
    - YAML: `yaml.safe_dump(sort_keys=True)` でキー順序を統一
    - CSV: `csv.writer` で再書き出し
    - TOML: `tomllib.loads()` → JSON化して比較

    ### ④ 余計な文章付加チェック

    構造化データ以外の説明文混入を検出・除外します。

    - **説明文パターン**: `"Here is..."`, `"Sure, ..."`, `"Output:"`, `"Result:"` 等
    - **コードフェンス境界**: `` ```json ... ``` `` の外にテキストがある場合を検出
    - **フォーマット境界**: JSONが `{`/`[` で始まらない、XMLが `<` で始まらない等

    ### ⑤ 外れ値・整合性チェック

    **外れ値検出（p99閾値）**
    - 文字数・行数・構造キー数の99パーセンタイルを算出
    - p99超のレコードを除外（極端に長い/複雑なサンプルが学習を不安定化させるのを防ぐ）
    - 閾値はデータから自動算出されるため、ハードコードされた定数に依存しない

    **整合性チェック（B1〜B8）**
    - メッセージ構造の検証（role/contentの存在、assistant応答の有無）
    - 制御文字の検出
    - daichira系のschema/format整合性チェック

    ### ⑥ CoT除去（Qwen3対応）

    u-10bei系データの `Approach: ... Output:` 形式のCoT（Chain-of-Thought）を除去し、`Output:` 以降の構造化データ本体のみを残します。

    **なぜ必要か**: Qwen3-Instructはネイティブで `<think>...</think>` 形式の思考トークンを使います。学習データに `Approach:/Output:` 形式のCoTが含まれると、モデルの事前学習パターンと衝突し、推論時に `<think>` や `<tool_call>` などの特殊トークンが出力に漏洩する原因になります。

    - CoT除去後、フォーマット別バリデータで構文再検証を実施
    - 除去後に空になったレコードは除外
    - 学習コード側では `SFT_MASK_COT = "0"` を設定（マスク不要）

    ---

    ## 意識した点

    ### 1. 前処理の重要性

    LLMの学習では「モデルアーキテクチャやハイパーパラメータの調整」に注目しがちですが、**データの前処理（クリーニング）がスコアに直結する**ことを実感しました。実際、9データセットをそのまま結合して高い学習率で学習した結果、特殊トークン漏洩・繰り返しループ・中国語出力混入などの深刻な品質劣化が発生しました。クリーニング後のデータ＋適切なハイパラで学習することで、これらの問題は完全に解消しました。

    ### 2. 原理に基づいた実装

    各フォーマットのバリデーションは「なんとなくパースできるか」ではなく、RFC/仕様書で定義されたルールを明示的にチェックしています。例えば：

    - CSVのRFC 4180では「ダブルクォート内のダブルクォートは `""` でエスケープする」というルールがあり、これをC4として独立にチェック
    - JSONのRFC 8259では `NaN`/`Infinity` が禁止されており、`parse_constant` フックで検出
    - YAMLの重複キー検出には `SafeLoader` のカスタムコンストラクタを使用し、ネストされた全レベルで検出

    こうすることで、「なぜこのレコードが落ちたのか」をエラーコード（C3, J4, Y2等）で正確に追跡できます。

    ### 3. LLM未使用の機械的処理

    コンペのルールに従い、データの生成・改変にLLMは一切使っていません。全ての処理はPython標準ライブラリおよび公開パッケージ（`json`, `csv`, `xml.etree`, `yaml`, `tomllib`, `hashlib` 等）による機械的な変換・バリデーションのみです。唯一 `transformers` の `AutoTokenizer` を使用していますが、これはトークン数のカウント（統計情報の算出）のみで、モデル推論は行っていません。

LLMの微調整（Fine-tuning）において、特定のフォーマット（JSONやYAMLなど）を遵守させつつ、モデル本来の推論能力を最大限に引き出すことは非常に繊細な作業です。Qwen3-4Bのような小型かつ高性能なモデルを対象に、検証スコアを段階的に引き上げるための実戦的なノウハウになります。
1. 基礎フェーズ：構造化データ出力の「安定化」
初期段階で最も重要なのは、回答が途切れることによる構文エラーを防ぐことです。

コンテキスト長の拡張: 複雑な構造化データを出力させる場合、標準的な長さでは不十分です。最大トークン数を拡張し、回答が最後まで完結する「器」を用意することが、スコア0.7の壁を超えるための第一歩です。
学習率のベースライン: モデルを破壊せず、かつ新しいルールを定着させるためには、低すぎず高すぎない「中道」の学習率設定が不可欠です。
2. 向上フェーズ：学習の「質」への転換
「型」を覚えた後のステップでは、モデルが生成する回答の正確性を高めることに注力します。

バッチ処理による勾配の純化: 1回の重み更新（ステップ）に使用するサンプル数を増やす（勾配累積）ことで、学習のノイズを抑えます。これにより、モデルは「たまたま正解したパターン」ではなく、より本質的なルールを抽出できるようになります。
「量」より「時間」の管理: 闇雲に全データを何度も見せる（エポック数を増やす）よりも、1回ごとの更新の精度を高めることが、特に小規模モデルの汎化性能を支えます。
3. 究極フェーズ：過学習（Overfitting）の抑制
高スコアを狙う上で最大の敵は、学習データの「丸暗記」です。

能動的なブレーキ（正則化）の導入: ドロップアウトやウェイトディケイを適切に設定し、モデルにあえて「負荷」をかけます。これにより、モデルが特定の文字列を暗記するのを防ぎ、未知の問題に対しても推論能力を発揮できる「柔軟な知能」を維持させます。
学習率の極微細なチューニング: ある程度のスコアに到達した後は、学習率を極めて微細な幅で増減させることが、モデルの潜在能力を100%引き出すスイートスポットの発見に繋がります。
結論：モデルとの対話としてのチューニング
LLMの微調整において、学習損失（Loss）の低下は必ずしも実力の向上を意味しません。数値上の改善と実際の推論精度のバランスを常に監視し、適切な「ブレーキ（正則化）」と「正確な更新（バッチサイズ）」を組み合わせること。このバランス感覚こそが、モデルを単なる「模倣者」から「真の推論者」へと進化させる鍵となります。
