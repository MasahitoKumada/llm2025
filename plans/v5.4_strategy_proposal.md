# v5.4 戦略提案ドキュメント

## 背景

v5.3の失敗分析から、以下の教訓が得られた:
1. **Empty Think Injection**は説明文付き出力を誘発し、純粋なデータフォーマットタスクに不適
2. **ハイパーパラメータの過激な変更**（特にLR 6倍増加）は既存知識を破壊
3. **ターゲットサンプルの追加**は適切なLRなしには逆効果

これらを踏まえ、v5.4では**v5.2の成功をベースに慎重な改善**を行う。

---

## 推奨戦略: Strategy A（v5.2ベース + 低LRターゲット学習）

### 概要

```
アプローチ: v5.2の学習済みモデルに対し、ターゲットサンプルのみで追加SFT
期待効果: v5.2の94.7%を維持しつつ、TOMLの7件エラーを改善
リスク: 低（既存の良いモデルを破壊しない）
```

### 実装詳細

#### Step 1: v5.2モデルの取得
```python
# v5.2の学習済みLoRAアダプターを使用
base_model = "deepseek-ai/DeepSeek-R1-Distill-Qwen-14B"
lora_adapter = "outputs/sft_v5.2/checkpoint-final"
```

#### Step 2: ターゲットサンプルの準備
```python
# ターゲットサンプル17件（Empty Think Injectionなし）
# - TOML: 14件
# - XML: 3件
target_samples_path = "inputs/sft_processed/v5.3_targeted_samples/targeted_training_samples.json"

# Empty Think Injectionを除去したバージョンを使用
# 各サンプルの出力は純粋なフォーマットデータのみ
```

#### Step 3: 追加SFTの実行
```python
# 非常に低いLRで追加学習
training_args = {
    "learning_rate": 1e-6,  # v5.3の3e-5の1/30
    "num_train_epochs": 1,  # 少ないエポック
    "per_device_train_batch_size": 1,
    "gradient_accumulation_steps": 4,
}

# LoRA設定はv5.2と同じ
lora_config = {
    "r": 64,
    "lora_alpha": 128,
    "target_modules": ["q_proj", "k_proj", "v_proj", "o_proj"],
}
```

### 期待される結果

| Format | v5.2 | 予測v5.4 | 根拠 |
|--------|------|----------|------|
| CSV | 100% | 100% | 変化なし（すでに完璧） |
| JSON | 100% | 100% | 変化なし |
| TOML | 72% | **80-85%** | ターゲットサンプルで2-3件改善 |
| XML | 95% | **100%** | ターゲットサンプルで1件改善 |
| YAML | 100% | 100% | 変化なし |
| **Total** | 94.7% | **96-98%** | +2-5件改善 |

### リスクと対策

| リスク | 確率 | 対策 |
|--------|------|------|
| 既存パフォーマンス低下 | 低 | 非常に低いLRで破壊的更新を防止 |
| ターゲットサンプル過学習 | 中 | 1エポックのみ、早期停止 |
| 効果なし | 中 | v5.2そのままでも94.7%は確保 |

## 代替戦略: Strategy B（v5.2設定 + データセット強化）

### 概要

```
アプローチ: v5.2のハイパーパラメータでゼロから学習、データセットにターゲットサンプル追加
期待効果: より安定した学習、TOMLの改善
リスク: 中（学習の再現性）
```

### 実装詳細

#### データセット構成
```python
# v5データセット（3,869件）+ ターゲットサンプル（17件）
# Empty Think Injectionは適用しない
total_samples = 3886

# フォーマット分布
format_distribution = {
    "JSON": 1500,  # 約39%
    "YAML": 1200,  # 約31%
    "CSV": 600,    # 約15%
    "XML": 400,    # 約10%
    "TOML": 186,   # 約5%（+14ターゲット）
}
```

#### ハイパーパラメータ（v5.2と同一）
```python
training_args = {
    "learning_rate": 5e-6,
    "num_train_epochs": 3,
    "per_device_train_batch_size": 1,
    "gradient_accumulation_steps": 8,
}

lora_config = {
    "r": 64,
    "lora_alpha": 128,
}
```

### 期待される結果

| Format | v5.2 | 予測v5.4 | 根拠 |
|--------|------|----------|------|
| CSV | 100% | 100% | 同等 |
| JSON | 100% | 100% | 同等 |
| TOML | 72% | **76-80%** | ターゲットサンプルで1-2件改善 |
| XML | 95% | **95-100%** | 同等または改善 |
| YAML | 100% | 100% | 同等 |
| **Total** | 94.7% | **95-97%** | +1-3件改善 |

### メリット・デメリット

**メリット**:
- v5.2の成功を確実に再現
- Empty Think Injectionの副作用を回避
- 学習過程がシンプル

**デメリット**:
- 計算コストが高い（フルトレーニング）
- ターゲットサンプルの効果が薄まる可能性

## 代替戦略: Strategy C（段階的Sequential Learning）

### 概要

```
アプローチ: フォーマット別に段階的にSFT、各段階でモデルをマージ
期待効果: 各フォーマットに特化した学習が可能
リスク: 高（複雑な手順、catastrophic forgetting）
```

### 実装詳細

#### Stage 1: JSON + CSV（最も多いフォーマット）
```python
# 2,100件で学習
stage1_data = json_samples + csv_samples
stage1_model = train(base_model, stage1_data, lr=5e-6)
```

#### Stage 2: YAML
```python
# stage1モデルをベースに1,200件で追加学習
stage2_model = train(stage1_model, yaml_samples, lr=2e-6)
```

#### Stage 3: XML + TOML（少数フォーマット）
```python
# stage2モデルをベースに600件で追加学習
# ターゲットサンプル17件を含む
stage3_model = train(stage2_model, xml_toml_samples, lr=1e-6)
```

### 期待される結果

| Format | v5.2 | 予測v5.4 | リスク |
|--------|------|----------|--------|
| CSV | 100% | 95-100% | 後段学習で忘却の可能性 |
| JSON | 100% | 95-100% | 後段学習で忘却の可能性 |
| TOML | 72% | **80-90%** | 専用学習で大幅改善期待 |
| XML | 95% | **95-100%** | 専用学習で改善期待 |
| YAML | 100% | 98-100% | 後段学習で微減の可能性 |

### メリット・デメリット

**メリット**:
- 少数フォーマット（TOML/XML）に集中的に学習可能
- 段階的アプローチで問題の切り分けが容易

**デメリット**:
- **Catastrophic forgetting**のリスク（特にStage1のJSON/CSV）
- 複雑なパイプライン
- 各段階のチューニングが必要
- v8シリーズで既に試行済み、結果は芳しくなかった

## 推奨: Strategy A

### 選定理由

1. **リスク最小化**: v5.2の94.7%を確実に維持しつつ改善を狙える
2. **計算効率**: 17件のみの追加学習で済む
3. **問題の明確化**: 効果がなければターゲットサンプル自体の問題と判明
4. **v5.3の教訓を活かす**: Empty Think Injectionを除去、超低LRを使用

### 実装チェックリスト

- [ ] v5.2の学習済みLoRAアダプターの確認
- [ ] ターゲットサンプルからEmpty Think Injectionを除去
- [ ] 追加SFT用のノートブック作成（v5.4.ipynb）
- [ ] ハイパーパラメータ設定（LR: 1e-6, epoch: 1）
- [ ] ローカル評価の実行
- [ ] LB提出と結果確認

### フォールバックプラン

Strategy Aで改善が見られない場合:
1. **Option 1**: v5.2をそのまま提出（94.7%は確保）
2. **Option 2**: Strategy Bを試行（フルトレーニング）
3. **Option 3**: ターゲットサンプルの再設計（別のエラーパターンに注目）

## 補足: v5.2追加学習の技術的実現可能性

### Kaggle環境での制約

```python
# v5.2のモデルはKaggle上に存在
# 追加学習には以下の選択肢がある:

# Option 1: Kaggleノートブックで追加学習
# - v5.2の出力を読み込み、追加SFTを実行
# - GPUメモリ制約あり

# Option 2: ローカルでの追加学習
# - v5.2のチェックポイントをダウンロード
# - ローカルで追加SFT
# - 結果をKaggleにアップロード

# Option 3: 新規データセットでゼロから学習（Strategy B）
# - 最も確実だが計算コスト高い
```

### 推奨実装方法

```python
# Kaggleノートブック内での追加学習
# 1. v5.2のLoRAアダプターを読み込み
# 2. ターゲットサンプル17件で追加SFT
# 3. マージして推論

from peft import PeftModel
from transformers import AutoModelForCausalLM

# v5.2モデル読み込み
base_model = AutoModelForCausalLM.from_pretrained(base_model_id)
model = PeftModel.from_pretrained(base_model, v5_2_adapter_path)

# 追加SFTの実行
trainer = SFTTrainer(
    model=model,
    train_dataset=target_samples,
    args=TrainingArguments(
        learning_rate=1e-6,
        num_train_epochs=1,
        # ...
    )
)
trainer.train()
```

## 結論

v5.4では**Strategy A（v5.2ベース + 低LRターゲット学習）**を推奨する。

この戦略は:
- v5.3の失敗から学んだ教訓を反映
- 既存の高性能を維持しつつ改善を狙う
- リスクを最小化しながら着実な進歩を目指す

実装に移行する際は、まずv5.2のモデル/チェックポイントの取得可能性を確認し、追加学習環境を整備することから始める。
