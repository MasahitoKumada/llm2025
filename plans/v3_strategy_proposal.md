## 2. 問題分析

### 2.1 TOMLフォーマット失敗の原因

v2で発生したTOMLエラーの例：
```
1. Key name found without value. Reached end of line.
2. What? list already exists?
3. Invalid inline table encountered
```

**推測される原因**:
1. ネスト構造の複雑なTOMLを正しく生成できていない
2. 配列（`[[array]]`）の構文が不正確
3. インラインテーブル（`{key = value}`）の使い方が不適切

### 2.2 SFTデータセットの問題点

#### データ分布の偏り
| Format | テスト比率 | SFT比率 | 差分 |
|---|---|---|---|
| JSON | 33.3% | 14.6% | -18.7% ⬇️ |
| YAML | 23.3% | 12.9% | -10.4% ⬇️ |
| TOML | 16.7% | 11.7% | -5.0% |
| XML | 13.3% | 13.6% | +0.2% |
| CSV | 13.3% | 11.5% | -1.9% |

**問題**: JSON/YAMLの比率がテストデータより低い

#### メタデータ不明データの多さ
- **unknown formatデータ: 12,000件（36%）**
- これらのデータの品質・内容が不明

## 3. v3 戦略提案

### 3.1 🎯 最優先: Text to TOML 強化

**目標**: TOML精度を60% → 90%以上に改善

#### 具体的アクション
1. **TOMLデータの増強**
   - Text to TOML のデータを500件以上追加
   - 複雑なネスト構造を含むサンプルを重点追加
   - 配列テーブル（`[[array]]`）のサンプルを増やす

2. **TOMLデータの品質向上**
   - 既存TOMLデータの検証と修正
   - 不正なTOML出力を含むサンプルの除去

3. **プロンプト/システムメッセージの改善**
   - TOML構文の厳密なルールを明示
   - よくあるエラーパターンの回避を指示

### 3.2 📊 データ比率の調整

**目標**: テストデータの分布に合わせたSFTデータ比率

#### 推奨データ構成（3,000件想定）
| Format | 目標比率 | 目標件数 |
|---|---|---|
| JSON | 33% | 990 |
| YAML | 23% | 690 |
| TOML | 20% | 600 (強化) |
| XML | 13% | 390 |
| CSV | 11% | 330 |

### 3.3 🔄 データキュレーション

#### 増やすべきデータ
1. **Text to TOML**: +500件（複雑構造含む）
2. **Text to JSON**: +200件（テストデータに2件しかない）
3. **CSV to JSON**: +100件（v0で失敗していた）

#### 減らすべき/除外すべきデータ
1. **unknownメタデータのデータ**: 12,000件を精査
   - 品質の低いデータは除外
   - メタデータを付与して再利用

2. **精度100%カテゴリの過剰データ**
   - JSON to YAML, TOML to JSON 等は現状維持でOK

### 3.4 🛠️ その他の改善

1. **出力フォーマットの統一**
   - 余計な説明文を含まないように訓練
   - コードフェンス有無の一貫性

2. **エッジケース対応**
   - 空配列、null値、特殊文字を含むデータ追加
   - Unicode文字（日本語等）を含むサンプル
