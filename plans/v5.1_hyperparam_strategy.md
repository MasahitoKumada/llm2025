### v5の問題点

#### WandBグラフからの観察
1. **train/loss**: 1.8→0.8まで下降後、最後に1.1まで上昇（過学習の兆候）
2. **eval/loss**: 1.25→0.85まで順調に下降
3. **grad_norm**: 初期は不安定（5）、step 100以降で安定（1付近）

#### v2 vs v5 出力比較
| 指標 | v2 | v5 | 評価 |
|---|---|---|---|
| 平均出力長 | 756文字 | 876文字 | ❌ v5が冗長 |
| 最大出力長 | 4,403文字 | 11,295文字 | ❌ v5が過剰 |
| コードフェンス | 9件 | 2件 | ✅ v5が改善 |
| 前置き文 | 4件 | 0件 | ✅ v5が改善 |
| 完全一致率 | - | 48.7% | - |

#### タスク別の問題
Generationタスク（Text to X）で出力が大幅に長くなっている：
- Text to XML: v2=1,073文字 → v5=2,971文字（+177%）
- Text to CSV: v2=1,486文字 → v5=2,435文字（+64%）
- Text to YAML: v2=1,110文字 → v5=2,164文字（+95%）

## 問題の仮説

### 主要仮説: Epoch=2による過学習
- 2エポック目で過学習し、冗長な出力を生成するようになった
- train/lossが最後に上昇しているのがその証拠
- Generationタスクで特に顕著（訓練データのパターンを過度に記憶）

### 副次仮説
1. **XMLエラー除去の影響は小さい**（64件、1.6%のみ）
2. **コードフェンス/前置き文の改善は効果あり**（プロンプト改良が機能）

## v6への提案

### 【推奨】Case A: Epoch=1に戻す

v2の設定に戻し、データのみv5（XMLエラー除去済み）を使用。

```python
os.environ["SFT_EPOCHS"] = "1"  # v2と同じ
os.environ["SFT_DATASET_ID"] = "LOCAL:/content/v5_train.json"  # XMLエラー除去済み
os.environ["SFT_LR"] = "5e-6"  # v2と同じ
os.environ["SFT_MAX_SEQ_LEN"] = "1024"  # v2と同じ
```

**期待効果**: 0.75以上（v2同等以上）

### 【オプション】Case B: Learning Rate調整

Epoch=1のままLearning Rateを下げる。

```python
os.environ["SFT_EPOCHS"] = "1"
os.environ["SFT_LR"] = "2e-6"  # 5e-6 → 2e-6に下げる
```

**根拠**: grad_normが初期に不安定だったため、より慎重な学習が有効かも

### 【オプション】Case C: Warmup比率増加

```python
os.environ["SFT_WARMUP_RATIO"] = "0.15"  # 10% → 15%
```

**根拠**: 初期の不安定性を軽減

### 【非推奨】以下のオプション
- **Epoch=3以上**: 過学習リスクが高すぎる
- **データ増加**: v3で失敗済み
- **データ大幅削減**: v4.1で失敗済み
