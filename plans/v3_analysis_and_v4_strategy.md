## v3分析結果

### 1. 品質指標は改善している

| 問題タイプ | v2 | v3 | 変化 |
|-----------|----|----|------|
| マークダウン囲み | 9件 | 2件 | ✅ 78%減少 |
| 説明文混入 | 5件 | 1件 | ✅ 80%減少 |

**矛盾**: 品質問題は減少しているのにスコアが下がった

### 2. タスクタイプ別の出力長変化

| タスクタイプ | v2 | v3 | 変化率 | 問題レベル |
|-------------|----|----|--------|-----------|
| Text to CSV | 1,486文字 | 3,052文字 | **+105.3%** | 🔴 深刻 |
| Text to YAML | 1,110文字 | 1,611文字 | +45.1% | 🟡 注意 |
| CSV to JSON | 886文字 | 1,098文字 | +23.8% | 🟡 注意 |
| CSV to XML | 1,076文字 | 687文字 | -36.1% | 🟡 注意 |
| YAML to XML | 831文字 | 484文字 | -41.8% | 🟡 注意 |

### 3. 致命的な問題パターン

#### 🔴 問題1: 無限ループ的な出力（Text to CSV）

**サンプル（p_723af76cde6edebcdae75167）**:
```
v3出力の問題:
...diet.primary_foods
...meat,meat,meat,meat,meat,meat,meat,meat,meat,meat,meat,meat...
```

同じ値が無限に繰り返される現象が発生。これはモデルが終了条件を正しく学習できていないことを示す。

#### 🔴 問題2: 誤認識による拒否（YAML to XML）

**サンプル（p_5a0b04abbc3900e01b2960a2）**:
```
v3出力:
"It looks like you've provided an XML snippet, but your request is to convert a YAML code to XML..."
```

正しいYAMLを誤ってXMLと認識し、変換を拒否している。

#### 🔴 問題3: 過少出力（Text to CSV）

**サンプル（p_eff14a8d30434a72c11a95ea）**:
```
v2: 4,403文字（複数行のCSV）
v3: 357文字（1行のみ）
```

v2では複数のサンプルデータを生成していたが、v3では1行しか出力しない。

### 4. 根本原因の推定

1. **データセット構成の変更が裏目に**
   - Text to TOMLを重視した結果、他のタスクの品質が低下
   - 特にText to CSV, YAML to XMLで顕著な悪化

2. **生成の終了条件の学習不足**
   - 繰り返し出力が発生（meat,meat,meat...）
   - 適切な長さで出力を終了できない

3. **入力形式の認識エラー**
   - YAMLをXMLと誤認識するケースが発生
   - 変換タスクで「これは○○形式ではありません」と拒否

## v4戦略

### 基本方針

**「v2をベースに最小限の改善を行う」**

v3の失敗から学んだ教訓：
- 大幅なデータセット変更はリスクが高い
- 品質問題の減少がスコア向上に直結しない
- 特定タスクの最適化が他タスクに悪影響を与える

### 具体的なアプローチ

#### 1. データセット構成

```
v4_dataset = v2のデータセット + 最小限の追加/修正
```

**使用データセット案**:
- ベース: v2で使用したデータセットをそのまま使用
- 追加: なし（または非常に少量のみ）
- 除外: 繰り返し出力を誘発するようなサンプルがあれば除外

#### 2. 品質フィルタリング（慎重に）

v3では過度なフィルタリングが悪影響を与えた可能性があるため：
- マークダウン除去: 行わない（v2でもスコアに影響なし）
- 説明文除去: 行わない
- 既存データをそのまま使用

#### 3. ハイパーパラメータ調整

v2と同じ設定を維持、または：
- エポック数を微調整（過学習を避ける）
- 学習率を微調整

#### 4. 検証プロセス

学習前に小規模なテストを行う：
1. v2のデータセットで学習
2. 少数のテストケースで推論
3. 繰り返し出力や誤認識がないか確認

### リスク管理

| リスク | 対策 |
|--------|------|
| v2と同じスコアになる | 許容（悪化よりマシ） |
| 繰り返し出力が発生 | 生成時のrepetition_penalty調整 |
| 特定タスクで悪化 | タスクタイプ別のテスト実施 |

## 結論

v3の失敗は「良かれと思った変更が裏目に出た」典型例。

v4では：
1. **保守的アプローチ**: v2の成功を維持
2. **最小限の変更**: 必要最小限の調整のみ
3. **慎重な検証**: 変更前後の品質確認

目標スコア: **0.75以上**（v2と同等以上）
