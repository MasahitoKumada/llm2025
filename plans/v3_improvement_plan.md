## 2. v3戦略: データ増減の具体的提案

### 2.1 【最優先】追加すべきデータ

#### ★ Text to TOML（最重要）
- テストデータ: 25件（最大割合）
- 現SFTデータ: 0件
- **推奨**: 500〜1000件追加

テキスト説明からTOML形式への変換パターン:
```
Input: "The application has name='MyApp', version='1.0', and debug mode is enabled..."
Output: 
name = "MyApp"
version = "1.0"
debug = true
```

#### ★ Text to その他形式
| タスク | テスト | 現SFT | 推奨追加 |
|--------|--------|-------|----------|
| Text to XML | 6 | 0 | 200〜300件 |
| Text to CSV | 6 | 0 | 200〜300件 |
| Text to JSON | 2 | 0 | 100〜200件 |
| Text to YAML | 1 | 0 | 50〜100件 |

#### ★ TOML関連タスク
| タスク | テスト | 現SFT | 推奨追加 |
|--------|--------|-------|----------|
| TOML to JSON | 11 | 18 | 300〜500件 |
| TOML to YAML | 8 | 28 | 200〜400件 |

#### ★ その他不足タスク
| タスク | テスト | 現SFT | 推奨追加 |
|--------|--------|-------|----------|
| CSV to JSON | 14 | 61 | 300〜500件 |
| JSON to YAML | 14 | 75 | 300〜500件 |
| YAML to JSON | 10 | 77 | 200〜400件 |
| XML to JSON | 13 | 377 | 200〜300件 |

### 2.2 削減/品質改善すべきデータ

#### 除去対象
1. **コードブロックを含む出力** (```)
   - 例: ` ```json\n{...}\n``` `
   - 正しい形式: `{...}`

2. **説明テキストを含む出力**
   - 例: "Here's the converted JSON:"
   - 例: "Let me convert this for you..."

3. **不要な思考プロセス**
   - 例: "First, I need to parse the CSV..."

#### 保持すべきパターン
- `Approach:` + `Output:` の構造は有効（v2で効果あり）
- 純粋な構造化出力のみを含むデータ

### 2.3 品質基準

新規データは以下を満たすこと:

1. **出力形式の純粋性**
   - JSON: `{` または `[` で開始
   - YAML: キー名または `-` で開始
   - TOML: `[section]` またはキー名で開始
   - XML: `<?xml` または `<root>` で開始
   - CSV: ヘッダー行で開始

2. **構造の複雑さ**
   - ネスト深度: 2〜4階層
   - 配列を含む構造
   - 特殊文字のエスケープ処理

3. **データ多様性**
   - 異なるドメイン（学生、製品、設定ファイル等）
   - 異なる配列サイズ（1〜10件）
   - 異なるフィールド数（5〜20フィールド）

## 4. 期待される効果

### 改善予測

| 項目 | v2 | v3（予測） |
|------|-----|----------|
| スコア | 0.75074 | 0.80+ |
| Text to X 正解率 | 低い | 大幅改善 |

### リスク

1. **過学習リスク**: 特定パターンへの過度な適合
   - 対策: データの多様性確保

2. **バランス崩れリスク**: 新タスク追加で既存タスクの精度低下
   - 対策: 既存の良好なデータも維持
