### 1.2 具体的なエラー例

#### パターン1: 複数行inline table (4件)
**問題**: TOMLのinline table `{ }` は一行で完結する必要があるが、モデルは改行を入れている

```toml
# ❌ モデルの出力（エラー）
exhibits = [
  {
    title = "Echoes of the Horizon"
    artist = "Liam Chen"
  }
]

# ✅ 正しいTOML
exhibits = [{ title = "Echoes of the Horizon", artist = "Liam Chen" }]
```

#### パターン2: 重複キー定義 (1件)
**問題**: 配列の記法 `[[...]]` を使わず、同じテーブル名を重複定義

```toml
# ❌ モデルの出力（エラー）
[artifact.inscriptions.list]
language = "Latin"

[artifact.inscriptions.list]  # 重複エラー
language = "Greek"

# ✅ 正しいTOML
[[artifact.inscriptions.list]]
language = "Latin"

[[artifact.inscriptions.list]]
language = "Greek"
```

#### パターン3: サブテーブル記法誤り (2件)
**問題**: `key =` の後に値なしで改行、YAML風のインデント構造を使用

```toml
# ❌ モデルの出力（エラー）
[planet]
name = "TOI-1788 b"
discovery =
  { year = 2023, method = "transit" }

# ✅ 正しいTOML
[planet]
name = "TOI-1788 b"
discovery = { year = 2023, method = "transit" }
```

## 3. v5.4ターゲット学習が効果なしだった理由

### 3.1 データ量の不均衡

| データ種別 | 件数 | 割合 |
|-----------|------|------|
| ベースデータ (v5) | 3,869 | 99.6% |
| ターゲットサンプル | 17 | 0.4% |

17件のターゲットサンプルは、3,869件のベースデータに埋もれてしまい、影響力がほぼゼロ。

### 3.2 学習率が低すぎた

| バージョン | Learning Rate |
|-----------|---------------|
| v5.2 | 5e-6 |
| v5.4 | 1e-6 |

v5.4はLR=1e-6という非常に低い学習率を使用。新しいパターンを学習するには不十分。

### 3.3 根本的なアプローチの問題

**Person Tの重要な発見**:
> "TOMLデータを削除してSFT学習しても、TOML: 21/25 = 84.0%"
> "TOMLの学習はTOMLデータから来ていない"

**意味**:
- モデルはTOML構文を**既に知っている**（ベースモデルの事前学習から）
- 問題は「正しいTOML構文を**いつ使うか**」という**判断**
- 少数のTOMLサンプルを追加しても、判断の改善にはつながらない

### 3.4 ターゲットサンプルの構造的限界

ターゲットサンプルで教えた内容:
1. inline tableは1行で書く
2. 配列テーブルは `[[table]]` 構文を使う
3. ネスト構造は `[table.subtable]` を使う

**しかし**:
- これらはモデルが既に知っている構文
- 問題は「複雑な入力に対して、どの構文を選択するか」
- 17件のサンプルでは、判断パターンの多様性が不足

## 4. Person Uの成功アプローチ (LB 0.84)

### 4.1 最終結果

| Format | Rate | Count |
|--------|------|-------|
| CSV | 100% | 20/20 |
| JSON | 100% | 50/50 |
| TOML | 92% | 23/25 |
| XML | 95% | 19/20 |
| YAML | 100% | 35/35 |

### 4.2 キーポイント

#### A. Sequential Learning (段階的学習)
> "ベースモデルに一度SFTを実行して、特定フォーマットの学習をします。
> そのモデルをベースにして、別のフォーマットの学習をします。"

1. Stage 1: フォーマットAで学習
2. Stage 2: 学習済みモデルをベースに、フォーマットBで学習
3. Stage 3: 繰り返し...

#### B. 最小限の学習
> "なるべくベースモデルに近い方が文章読解力が高くて安定した合格スコアになる"

- 学習しすぎない
- ベースモデルの能力を活かす

#### C. 学習状態の見極め
> "Here!とかSure!が出たら、もうちょっと学習できそう。
> 同じこと繰り返してるものが出たら、ちょっとやりすぎ"

- 自然言語混入 → 学習不足
- 繰り返し出力 → 過学習

### 4.3 各データセットの特性発見

> "フォーマットAのスコアがよくなっても、フォーマットBのスコアはいまいち"
> "いい感じで混ぜればよくなるか？というと、そうはならない"

**解決策**: 単純マージではなく、Sequential Learning

## 6. 結論

### v5.4が効果なしだった本質的理由

1. **アプローチの誤り**: ターゲットサンプル追加は、モデルが「既に知っている」ことの再学習
2. **データ量の不均衡**: 17件 vs 3,869件では影響力なし
3. **学習率が低すぎ**: LR=1e-6では新パターンを吸収できない
4. **根本原因の見誤り**: TOMLエラーは「構文知識の欠如」ではなく「判断の誤り」

### 0.8超えに必要なアプローチ

1. **Sequential Learning**: フォーマット別の段階的学習
2. **少量高品質データ**: 1000件以下の厳選データ
3. **最小限の学習**: ベースモデルの能力を維持
4. **学習状態の監視**: 過学習の早期検出
